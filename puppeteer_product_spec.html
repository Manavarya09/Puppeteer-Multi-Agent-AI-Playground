The core idea

Most AI tools give you one model doing one thing. Puppeteer gives you a team of specialist agents — a Planner, a Coder, a Critic, a Web Searcher — coordinated by a central AI orchestrator that learns over time which agents to call, in what order, and when to stop. The result is dramatically better answers at lower cost.

What makes it different Differentiation

Unlike AutoGPT, CrewAI, or LangGraph — which use static, human-designed agent pipelines — Puppeteer's orchestrator learns its own coordination strategy via reinforcement learning. The topology of agent interactions evolves: simpler at first, then converging into dense, cyclic graphs as it discovers which agents work best together for each task type.

This means no manual pipeline engineering. The system discovers optimal team structures on its own.

Core Features

P0 — MVP Task Playground

A clean, single-screen interface where users type a task in natural language. The system automatically selects agents, shows their outputs streaming in real time (like a group chat), and delivers a final synthesized answer. Users can see token usage, agent sequence, and overall confidence.

P0 — MVP Live Orchestration Graph

A real-time animated graph showing which agents were activated, in what order, and how they connected. Nodes pulse when active. Edge thickness represents interaction frequency. Users can toggle between "compact cyclic" (evolved) and "simple chain" (initial) views to understand how the system optimizes itself.

P0 — MVP Agent Transparency Panel

Each agent's full reasoning trace is visible. Users can expand any agent's output to see its raw prompt, the response it generated, and what the orchestrator decided to do next. This builds trust and is essential for researchers and power users.

P1 REST API + SDK

Developers call POST /v1/run with a task description and optional agent constraints. Response streams token-by-token with SSE. Includes a Python SDK and Node.js SDK with one-line setup. Rate limits and usage metered per API key.

P1 Custom Agent Builder

Teams users can define their own agents with a system prompt, tool set, and reasoning pattern. Custom agents are treated identically to built-in ones by the orchestrator. This enables domain-specific specialization — e.g. a "Legal Document Reviewer" agent or a "SQL Query Optimizer" agent.

P2 Topology Evolution Replay

For researchers: a scrubable timeline showing how agent interaction patterns evolved over multiple task episodes. Visualizes the transition from disjoint chains → trees → dense cyclic graphs. Exportable as JSON interaction logs for offline analysis.

P2 Benchmarking Mode

Users submit a task and ground truth answer. The platform runs the task through: (a) base LLM alone, (b) static multi-agent baseline, and (c) Puppeteer evolved. Returns accuracy, token usage, and cost comparison across all three. Designed for researchers and developers evaluating the approach.

System Architecture

High-level stack
Frontend — React + TypeScript · Real-time graph (Cytoscape.js) · Streaming UI
↕
API Gateway — FastAPI · Auth (JWT + API keys) · Rate limiting · SSE streaming
↕
Orchestration Engine — Puppeteer policy model · Agent registry · State manager · RL reward tracker
↕
Agent Pool — Containerized agents (Docker) · Tool sandboxes · LLM calls (OpenAI / Anthropic / local)
↕
Data Layer — PostgreSQL (tasks, users, logs) · Redis (session state) · S3 (interaction exports)
Request flow
1

User submits task

Frontend sends task string + optional constraints to POST /v1/run. Server assigns task ID and opens SSE stream.

2

Orchestrator initializes state

Puppeteer policy loads task embedding, prior episode statistics, and agent registry. Picks first agent based on task type classification.

3

Agent execution loop

Selected agent runs, appends to shared state S_t. Orchestrator evaluates state and selects next agent (or Terminator). Repeats until termination.

4

Aggregation + streaming

ConcluderAgent synthesizes final answer. F_agg combines outputs. Each agent output streams to frontend via SSE as it's produced.

5

Reward logging

Task outcome (if ground truth available) + token cost logged. RL policy updated asynchronously via REINFORCE to improve future orchestration.

Model backends

Titan subspace (high accuracy): GPT-4o, Claude Opus

Mimas subspace (cost efficient): Llama-3-8B, Claude Haiku

Users choose subspace per task — trading accuracy for cost.

Agent isolation

Each agent runs in a sandboxed container. Tool-use agents (Python, Browser) have network + filesystem limits. No cross-agent memory leakage by design.

Built-in Agent Library

The orchestrator selects from these agents dynamically. Each is independently versioned and can be upgraded without touching the orchestrator policy.

Teams users define agents via a simple JSON schema: name, system_prompt, tools[], reasoning_pattern, cost_weight. The orchestrator automatically incorporates custom agents into its selection policy after 5+ invocations of training data.

Risks & Mitigations

LLM API cost spiral
Multi-agent tasks use 5–15× tokens. Mitigate: strict token budgets per task, RL incentivizes early termination, Mimas (cheap) mode as default
Agent reliability / hallucination
CriticAgent + ReflectAgent catch errors in-loop. Show confidence scores. Never present agent output as verified fact without a source
Tool sandbox security
PythonAgent + BrowserAgent run in Docker with no internet/filesystem access beyond allow-listed domains. Code execution timeout: 30s
Orchestrator policy cold start
Ship with pre-trained weights from paper experiments. RL improves with usage. Fallback to rule-based orchestration if RL confidence is low
Latency (multi-agent is slow)
Stream intermediate outputs so users see progress immediately. Target <2s to first token. Set honest expectations: complex tasks take 30–120s
Competitor (AutoGPT, CrewAI)
Differentiate on: dynamic topology, RL optimization, research credibility, visualization quality. Focus on power users who've been burned by static pipelines
User trust / opacity
Full transparency panel is P0. Every agent decision is explainable. "Why did the orchestrator choose this agent?" as a clickable question
